  - 总结笔记/视频/评论区功能
  - 拍照识图、地图标点、语音输入

### 脚本规格要求
| 项目 | 要求 |
|------|------|
| 视频时长 | ≤ 1分钟 |
| 脚本字数 | ≤ 260字（口播文案） |
| 视频比例 | 9:16竖版 |
| ai大小写 | 必须小写：ai |
| 首次提及 | 必须同时出现"小红书"+"点点" |

| 必带话题 | #做攻略用点点 #当我问点点 #点点ai #小红书ai |

### 脚本结构（4模块）
1. **基础信息头**：标题、类目、画面建议
2. **开头痛点**：前5-8秒，具体可感知的痛点
3. **点点解决方案**：功能演示，必须包含录屏/截图
4. **价值验证+结尾**：callback点点的价值延伸

### 创作方向分类
- **旅行类**：本地周末Citywalk、旅行目的地决策、旅行行中真实信息避坑
- **本地类**：本地周末吃喝玩乐、本地周末亲子
- **泛生活类**：职场效率、生活经验干货、学习考试、美妆穿搭助手、运动入门教练

### 具体修改问题（根据实际交付总结）
1. **标题不够吸引**：多次优化标题，增加痛点关键词和情绪钩子
2. **开头痛点不够具体**：从泛泛"不知道怎么办"改为具体场景描述
3. **点点功能演示不够清晰**：补充录屏/截图的具体画面描述
4. **价值验证不够强**：增加使用点点的具体效果对比
5. **ai大小写问题**：部分脚本写成AI，全局改为ai

## 🧠 深度方法论反思（2026-02-12晚补充）

### 反思1：前期沟通的深层问题

**问题现象**：
- 甲方在过程中不断加限制条件
- 很多东西甲方自己也没想清楚
- 项目急，但沟通不充分就开工

**根因分析**：
1. **甲方视角**：甲方自己也还在探索需求，过程中不断明确
2. **我方视角**：看到需求文档就开始做，没有主动追问不清楚的点
3. **时间压力**：项目急（3天80条），压缩了沟通时间

**下次改进**：
- ❌ **不要**：拿到需求文档就急着开始做
- ✅ **应该**：列出所有疑问点，主动和甲方确认清楚再开工
  - "哪些场景不能做？"
  - "还有哪些限制条件没写清楚？"
  - "春节活动的具体要求是什么？"
  - "可以先确认1-3条的方向吗？"

---

### 反思2：框架先行 vs 批量生成

**错误做法（这次）**：
```
需求文档 → AI直接生成80条 → 一个个改 → 效率低
```

**问题**：
1. **生成策略错**：直接让AI生成一大堆脚本
2. **质量不可控**：长得差不多，不知道哪些好哪些坏
3. **修改成本高**：每条都要改，耗时巨大
4. **主题混乱**：有些行有些不行，整体很乱

**正确做法（下次）**：
```
需求文档 → 确定框架 → 确定场景 → 甲方确认 → 分批生成脚本
```

**关键改进**：

#### Step 1：框架先行（和数字分身一起）
- 确定脚本结构（哪几部分、每部分多长）
- 确定必加内容的位置（如春节活动在哪插入）
- 确定可变内容的位置（如具体痛点、具体解决方案）
- 确定必加内容的变种说法（春节活动有多少种表达方式）

#### Step 2：场景/话题先行（而非脚本先行）
**认知转变**：甲方在意的是**场景**，不是脚本本身
- 先列出一批场景/话题/痛点
- 跟甲方确认：哪些场景合适？哪些不合适？
- 确认后再基于这些场景去填充脚本

**为什么场景思维更对**：
- 甲方评审时关注：这个场景是否偏？这个痛点是否真实？
- 如果场景都不对，脚本写得再好也没用
- 先对齐场景，避免后期大规模返工

#### Step 3：分批次确认+生成
- 不要一次性生成80条
- 先生成10-15条，给甲方看一下
- 确认方向没问题后，再继续下一批
- 每批10-15条，及时调整

---

### 反思3：AI工具的正确使用方式

**错误认知**：AI = 批量生成器
- 让AI一次性生成80条
- 生成结果长得差不多，质量不可控

**正确认知**：AI = 结构化助手
- 用AI确定框架（哪些是固定模板，哪些是可变内容）
- 用AI生成场景矩阵（列出可能的场景）
- 用AI筛选场景（哪些合适，哪些不合适）
- 用AI填充已确认的脚本（基于场景+框架）

**AI的价值排序**（从高到低）：
1. **框架设计**：哪些是固定模板，哪些是变量
2. **场景生成**：列出可能的场景/话题矩阵
3. **场景筛选**：基于甲方要求筛选合适场景
4. **脚本填充**：基于确认的场景和框架，填充脚本内容
5. **批量生成**：仅在前面都确认后，才批量生成

### 核心方法论升级

#### 从"交付思维"到"产品思维"
**交付思维（错误）**：
- 拿到需求就做
- 做完就交付
- 交付就结束

**产品思维（正确）**：
- 先理解甲方真正在意什么（场景、痛点真实性）
- 建立可复用的框架（下次类似项目直接用）
- 每个项目都在积累方法论，而不只是完成交付

#### 从"脚本思维"到"场景思维"
**脚本思维（错误）**：
- 关注脚本本身的质量
- 一条一条写，每条都要改
- 低效重复劳动

**场景思维（正确）**：
- 关注场景是否对、痛点是否真实
- 先确定场景范围，再填充脚本
- 避免在错误场景上浪费时间

## 📚 通用方法论（适用所有项目）

### 原则1：框架先行
任何项目都不要急着做内容，先确定：
1. **结构框架**：哪些是固定部分，哪些是可变部分
2. **必加内容**：哪些内容必须出现，在什么位置
3. **检查清单**：硬性要求有哪些，如何检查

### 原则2：内容分层
不要一上来就写最终内容，分层确定：
1. **话题层**：哪些话题/场景可以做
2. **框架层**：每条脚本的结构是什么
3. **内容层**：基于确认的话题+框架填充具体内容

### 原则3：分批确认
不要一次性做完全部：
1. **小范围试错**：先做1-3条/10-15条
2. **及时确认**：给甲方确认方向
3. **迭代优化**：确认没问题后再继续下一批

### 原则4：AI作为辅助而非主力
- AI主力 = 做你的工作（错误，质量不可控）
- AI辅助 = 帮你结构化思考（正确，提高效率）

**AI的正确使用流程**：
1. 框架设计：和AI一起确定框架
2. 场景生成：让AI列出可能的场景
3. 内容筛选：和AI一起筛选哪些场景合适
4. 内容填充：基于确认的内容，用AI填充脚本
5. 质量检查：用AI帮助检查是否符合要求
